QA Copilot — Multi-Source RAG with a Built-in Explorer

A retrieval-augmented copilot for QA engineers, grounded in 5 heterogeneous artifacts: a Selenium framework, a Playwright framework, the VWO test-case corpus, product PDFs (PRDs), and JIRA bug exports. Hybrid retrieval (dense + sparse), cross-encoder rerank, intent routing, and a transparent RAG Explorer that exposes every chunk and every decision behind an answer.

01

Overview

Capstone

QA Copilot answers natural-language questions like "Show the BasePage waitForElement implementation", "List P0 Blocker tests for Admin", or "What does the PRD say about login dashboard auth?". Each answer cites the chunks it used. The RAG Explorer exposes the full pipeline so you can debug retrieval the way you debug code.

Why hybrid?

Dense vectors capture meaning; sparse vectors (BM25-like) capture rare tokens — JIRA ids, method names, status flags. Reciprocal Rank Fusion combines both. Code & ticket queries need this.

Why route?

A small LLM classifier picks 1–2 collections per query. "Show login fixture" → playwright_code only. "Open bugs about login" → vwo_bugs + vwo_testcases. Faster, less noise.

Why explore?

When an answer surprises you, inspect the retrieval. The Explorer tab shows the rewrite, the route decision, per-collection hits, RRF scores, rerank scores, the final context, and the LLM call.

03

Five Sources, Five Collections

Collection Source Chunk strategy Key metadata Complexity
selenium_code Selenium Java repo tree-sitter-java; one chunk per method or top-level class. path, package, symbol, kind, start_line, end_line Med
playwright_code Playwright TS repo tree-sitter-typescript; per fn/class + each test() block. path, kind, symbol, test_title, lines Med
vwo_testcases CSV · 100 TCs Row-aware: 1 TC = 1 chunk (sliding-window if > 1000 chars). tc_id, jira_id, module, priority, severity, sprint, status, owner Low
vwo_docs PDFs (PRDs) PyMuPDF page text → 800/120 sliding window. Skip pages < 50 chars. doc_title, page, source_path Low
vwo_bugs JIRA MD exports Parse JIRA header (Status / Priority / Reporter / …); chunk body. jira_id, status, priority, reporter, assignee, summary, created Low
Every collection uses named vectors — dense (1024d cosine) and sparse (BM25-style lexical weights). One Qdrant point holds both, retrieved independently and fused with RRF.
04

Query Pipeline · 6 Stages

STAGE 01
Rewrite
Condense follow-ups + last N turns into one standalone query (Groq).
STAGE 02
Route
Classifier picks 1–2 collections from the 5. UI can override.
STAGE 03
Hybrid Search
Dense + sparse query → top-K per collection → RRF fuse (k=60).
STAGE 04
Rerank
bge-reranker-v2-m3 cross-encoder over top-12 → top-4.
STAGE 05
Compose
Tag chunks <doc id="1" src="…"> + system prompt.
STAGE 06
Stream
Groq → SSE tokens. Inline [n] citations resolved client-side.

How RRF works (k=60)

def rrf(rank_lists, k=60):
    score = {}
    for ranked in rank_lists:
        for r, doc_id in enumerate(ranked, start=1):
            score[doc_id] = score.get(doc_id, 0.0) + 1.0 / (k + r)
    return sorted(score.items(), key=lambda t: t[1], reverse=True)

Each list contributes 1/(60+rank). A doc ranked #1 in two lists scores higher than a doc ranked #1 in one list and absent in the other — that's the rank-fusion intuition.

05

RAG Explorer — Debug Every Answer

Debugger

The Explorer is a second UI mode that runs the same query but returns the entire trace instead of streaming an answer. Every stage of the pipeline is rendered as an expandable card. Use it to answer: Why did the LLM say that? Which chunks did it actually see? Did the router pick the right collections?

What it shows, per query

PanelContentWhat you learn
① Query RewriteOriginal question · last N chat turns · rewritten standalone query.Whether history hurt or helped retrieval.
② Router DecisionClassifier output: [selenium_code, vwo_testcases] + a 1-line reason.Whether the right collections were searched.
③ Per-Collection HitsTop-K dense hits, top-K sparse hits, with payload preview & score.Which retrieval mode pulled which chunk.
④ RRF FusedCombined ranking, fused score, source ranks per list.How fusion changed the order.
⑤ RerankCross-encoder score per candidate, before/after order.Whether rerank rescued or hurt a chunk.
⑥ Final ContextThe exact text blocks sent to the LLM, tagged <doc id="…">.The ground truth the LLM actually saw.
⑦ LLM CallSystem prompt · user message · model · temperature · token usage.Exactly what was generated and on what basis.
⑧ Answer + CitationsStreamed answer with [n] chips that scroll source cards into view.End-to-end trace of how the answer maps to chunks.

Sample trace JSON shape

type Trace = {
  query: { original: string; history: Turn[]; rewritten: string };
  router: { collections: string[]; reason: string };
  per_collection: Record<string, {
    dense_hits: Hit[]; sparse_hits: Hit[]; fused: FusedHit[];
  }>;
  rerank: { input: FusedHit[]; output: RerankedHit[] };
  context: { blocks: { id: number; source: string; text: string }[] };
  llm: { model: string; temperature: number; system: string;
         user: string; usage: { prompt: number; completion: number }; };
  answer: { text: string; citations: number[] };
  timings_ms: Record<string, number>;
};
Practical use: if a question returns a wrong answer, switch to Explorer, replay it. 9 times out of 10 the bug is upstream of the LLM — wrong router pick, missing chunk, bad rerank.
06

Components

A

FastAPI Backend

Service Layer

Async HTTP. Endpoints: /api/chat (SSE), /api/explore, /api/health, /api/ingest/*. CORS enabled for the Vite dev server.

B

BGE-M3 Embedder

Embedding Model

Single model emits both dense (1024d) and sparse vectors. FlagEmbedding loader, fp16 by default, threaded singleton for hot reuse.

C

Qdrant Store

Vector DB

Embedded file mode by default (./qdrant_data). 5 collections, named-vectors schema. Optional HTTP server mode via QDRANT_URL.

D

tree-sitter Chunkers

Code Ingest

Java + TypeScript grammars produce per-symbol chunks with file path, package, line span, kind. Fallback skips unparsable files.

E

Row-Aware Chunker

Test-Case Ingest

Ported from Chapter 8 Advance RAG: 1 TC = 1 chunk, field-labeled, oversize rows fall back to overlap-windowed chunks.

F

PDF / JIRA Parser

Docs Ingest

PyMuPDF page-by-page, skip empty pages with a JSON skip report. JIRA MD parser extracts header fields into structured metadata.

G

Intent Router

Routing

One Groq classifier call returns selected collections + a short reason (used by the Explorer). UI can override.

H

Hybrid Retriever + RRF

Retrieval

Dense and sparse top-K per selected collection, fused via Reciprocal Rank Fusion (k=60). Returns top-12 candidates.

I

BGE Reranker v2-m3

Reranking

Cross-encoder scoring (question, candidate). Cuts top-12 to top-4 before answer composition. Lifts factual precision noticeably.

J

Groq · gpt-oss-120b

LLM

Streaming chat completion. Answer mode (cited) or generate mode (drafts a new TC from style exemplars).

K

React Chat UI

Frontend

Vite + TS + Tailwind. SSE consumer renders markdown, inline [n] chips scroll source cards. Multi-turn memory.

L

RAG Explorer Page

Debug UI

Replays any query and renders the full trace as expandable stage cards (rewrite → route → hits → fuse → rerank → context → LLM → answer).

Hourly Scheduler

APScheduler

A separate process re-indexes Qdrant on a fixed cadence (default every 60 minutes). Wraps the same five ingest_*.run() functions used by ingest_all.py, inside an APScheduler BlockingScheduler. No extra services — one Python process, configurable from .env.

Env knobs

VariableDefaultPurpose
INGEST_INTERVAL_MINUTES60Minutes between cycles in long-lived mode.
INGEST_AT_STARTUPtrueRun one cycle immediately, then loop.
INGEST_TARGETSselenium,playwright,testcases,pdfs,jiraComma-separated subset of pipelines.
INGEST_RECREATEfalseDrop & rebuild collections each cycle (expensive).
INGEST_LOCK_FILE./data/.ingest.lockSkips overlapping cycles via O_CREAT|O_EXCL.

Two run modes

# A) Long-lived in-process loop (foreground or backgrounded with nohup / systemd):
python -m scheduler.run

# B) One-shot for cron / launchd / systemd timers:
python -m scheduler.run --once

# Override targets for a single run:
INGEST_TARGETS=testcases,jira python -m scheduler.run --once

Linux systemd timer (recommended for VPS)

# /etc/systemd/system/qacopilot-ingest.service
[Unit]
Description=QA Copilot ingest one-shot
After=network-online.target

[Service]
Type=oneshot
WorkingDirectory=/opt/qacopilot/Chapter_09_Project_QACopilot
ExecStart=/opt/qacopilot/Chapter_09_Project_QACopilot/.venv/bin/python -m scheduler.run --once
User=qacopilot

# /etc/systemd/system/qacopilot-ingest.timer
[Unit]
Description=Run QA Copilot ingest every hour

[Timer]
OnBootSec=2min
OnUnitActiveSec=1h
Unit=qacopilot-ingest.service

[Install]
WantedBy=timers.target

# Enable
sudo systemctl daemon-reload
sudo systemctl enable --now qacopilot-ingest.timer
Single-process Qdrant gotcha. The default file-store is single-process — ingest running inside the API process would block requests; sibling processes will race on the file lock. Either run ingest only when the API is stopped, or switch both to HTTP mode by setting QDRANT_URL. See scheduler/README.md.

Operational notes

  • Secrets: store GROQ_API_KEY in .env (chmod 600), not in .env.example. For AWS/GCP use Secrets Manager / Secret Manager and inject via EnvironmentFile rendered at deploy time.
  • Backups: snapshot /opt/qacopilot/qdrant_storage nightly. Re-ingest is the disaster-recovery fallback (idempotent).
  • Updates: git pull && pip install -r backend/requirements.txt && npm run build && systemctl restart qacopilot-api.
  • Observability: journalctl -u qacopilot-api -f · tail -f data/_scheduler.log · UI /status page.
  • GPU instance (optional): AWS g5.xlarge / GCP n1-standard-4 + T4. Add nvidia-container-toolkit for Docker; install CUDA-enabled PyTorch wheel.
  • Container alternative: wrap FastAPI in a Dockerfile with the venv baked in; deploy to AWS ECS Fargate / GCP Cloud Run (1 vCPU min, set request timeout to 300s for SSE).

Design Trade-offs (so future-you remembers)

DecisionWhy thisWhat we gave up
5 collections, not 1Cleaner routing, smaller candidate sets, tunable per source.Have to maintain N ingest scripts.
BGE-M3 dense + sparseOne model, two retrieval modes. Sparse rescues code/JIRA-id queries dense often misses.~2GB first-load download, slower cold start.
Cross-encoder rerankBig precision lift on the top-4 the LLM actually sees.~150ms extra per query.
LLM router (not user dropdown)Zero-friction UX; explorer shows the decision so it's never magic.~150ms small Groq call per query.
Qdrant embedded modeNo Docker, no server. Single-binary setup for the chapter.One process at a time on the file store.
tree-sitter, not regexReal AST → reliable per-method chunks. Survives reformats.Adds two grammar wheels.
Pedagogy: this chapter is the capstone of the RAG track — Chapter 8 Basic (ChromaDB + Nomic) → Chapter 8 Advance (Qdrant + BGE-M3 hybrid) → Chapter 9 QA Copilot (multi-collection + router + explorer). Each step adds one production concern: hybrid retrieval, then rerank, then routing, then transparency.
