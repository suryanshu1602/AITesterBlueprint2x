An LLM-as-judge harness that scores two live apps — a support chatbot and a full RAG pipeline — on relevancy, faithfulness, grounding, hallucination, bias, toxicity and more. One metric registry, two ways to run it: pytest for CI and an interactive dashboard for demos.

01 · Why

§1The objective

Chapter 18 evaluated single prompts. Chapter 19 goes further: evaluate whole production-shaped systems. We stand up two real apps, then point a configurable judge LLM at them and measure quality the way a QA engineer would — repeatably, with thresholds, pass/fail, and a reason for every score.

Goal

Auditable quality gates

Turn "the bot feels good" into numbers: relevancy ≥ 0.7, hallucination ≤ 0.4, etc. — enforced in CI.

Goal

Swappable judge

Same metrics, judged by OpenAI, Groq, or a fully local Ollama model — flip one env var.

Goal

Two surfaces

One registry feeds both a pytest suite (CI) and a click-to-run web dashboard (teaching/demo).

03 · Flow

§3How the pieces connect

The framework (C) calls each app over HTTP, wraps the response in a DeepEval LLMTestCase, and hands it to a metric. The metric prompts the judge LLM, which returns a score + reason. First the map, then watch one metric run end-to-end.

Drawn with the Excalidraw MCP · apps under test → framework → judge → verdict App A App B Framework Judge Verdict

Now the loop in motion — one metric, one test case, scored by the judge and gated by a threshold. It cycles through real verdicts the framework has produced:

The bar fills to the judge's score; the tick is the threshold. Quality metrics pass when the score clears the line; safety metrics flip it — the score must stay under the ceiling.

Inside Subsystem BHow the RAG pipeline & vector DB work

The RAG Explorer does the work in two passes. Ingest runs once — split the docs, embed each chunk, write the vectors into ChromaDB. Query runs per request — embed the question, do a similarity search in the DB, hand the top-k chunks to the LLM, answer with citations.

Drawn with the Excalidraw MCP · one DB, two passes — ingest writes, query reads I/O Embed Vector DB LLM Answer

Inside a metricHow DeepEval actually scores a test case

Every metric ends at the same place — a number between 0 and 1 plus a written reason. A GEval metric hands the judge a plain-English rubric and lets it reason step by step (chain-of-thought) before grading; built-in metrics ship their own rubric. is_successful() then compares the score to the threshold in the right direction.

05 · Coverage

§5What we measure (the 22 metrics)

Direction matters: for quality metrics a higher score is better (threshold is a floor); for safety metrics a lower score is better (threshold is a ceiling). DeepEval handles the inversion via is_successful().

Drawn with the Excalidraw MCP · 6 families across 2 systems — the registry the dashboard renders from Quality Safety Retrieval G-Eval
Chatbot · quality

Relevancy · Faithfulness · Hallucination · Completeness (G-Eval)

Stays on topic, grounded in policy, no contradictions, covers every key fact.

Chatbot · safety

Bias · Toxicity · PII Leakage · No-Prompt-Leak (G-Eval)

Refuses adversarial / biased prompts; never reveals the system prompt.

Chatbot · conversational

Conversation Completeness · Knowledge Retention

Multi-turn: remembers earlier turns, satisfies intent across the dialog.

RAG · retrieval

Contextual Precision · Recall · Relevancy

Right chunks ranked first, nothing missing, little noise.

RAG · quality

Faithfulness · Answer Relevancy · Hallucination · Correctness (G-Eval)

Answer grounded in retrieved context and factually matches the golden.

RAG · G-Eval

Citation Quality · Helpfulness

Cites a real source filename; gives a specific, actionable answer.

07 · Run it

§7How to run the commands

Start both apps first (A on :8201/:5173, B on :8202), then pick a surface. .env is loaded automatically by conftest.py / the dashboard.

# from Chapter_19_DeepEval_Framework/03_DeepFramework
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt          # pins deepeval==3.9.9 + FastAPI

# .env (gitignored) — set keys + choose the judge
# GROQ_API_KEY=gsk_...   OPENAI_API_KEY=sk-...   JUDGE_PROVIDER=openai
# per-golden test cases, CI-style
export JUDGE_PROVIDER=openai            # or groq / ollama
pytest tests/chatbot/test_01_chatbot_answer_relevancy.py -v

# filter by marker
pytest -m "chatbot and quality" -v
pytest -m needs_rag -v                    # auto-skips if RAG is down
# same test, results pushed to the Confident AI cloud dashboard
export CONFIDENT_API_KEY=confident_...
deepeval test run tests/chatbot/test_01_chatbot_answer_relevancy.py

# tip: use the OpenAI judge here — Groq's free 8000 TPM cap
# gets exceeded when deepeval fans out judge calls.
# interactive: click any metric, watch live pass/fail/score/reason
uvicorn dashboard.app:app --port 8203
# open http://localhost:8203 — switch judge provider in the UI

# or hit the REST API directly
curl -X POST localhost:8203/api/run \
  -H 'Content-Type: application/json' \
  -d '{"metric_id":"rag.faithfulness","sample_idx":0}'

08 · Extend

§8How to add another metric

Because the registry is the single source of truth, adding a metric is mostly one row. It shows up in the dashboard instantly; the matching pytest file is optional.

  1. Pick or write the metric

    Use a built-in DeepEval metric, or define a GEval with a plain-English rubric. Add a tiny factory in registry.py:

    def _geval_conciseness(j, t):
    return GEval(
        name="Conciseness",
        criteria="Penalise answers that pad with facts the question didn't ask for.",
        evaluation_params=[LLMTestCaseParams.INPUT, LLMTestCaseParams.ACTUAL_OUTPUT],
        model=j, threshold=t,
    )

  2. Register one row

    Append a MetricDef to REGISTRY. higher_is_better sets the threshold direction; sample_kind tells the runner which dataset to pull (golden / safety / pii_probe / summary / conversation).

    MetricDef("chatbot.conciseness", "G-Eval · Conciseness",
    "Answer avoids unasked-for extras.", "geval", "chatbot",
    0.6, True, "golden", _geval_conciseness,
    ["input", "actual_output"]),

  3. (Optional) add a pytest file

    For CI + Confident AI capture, mirror it as tests/test_NN_*.py using assert_test so each golden becomes a captured test case:

    @pytest.mark.parametrize("golden", CHATBOT_GOLDENS, ids=lambda g: g.input[:45])
def test_conciseness(chatbot, judge, golden):
    reply = chatbot.chat(golden.input).reply
    tc = LLMTestCase(input=golden.input, actual_output=reply)
    assert_test(tc, [_geval_conciseness(judge, 0.6)])

  4. (If it needs new data) extend the goldens

    Add cases to datasets/chatbot_goldens.py or rag_goldens.py. Fill the fields the metric's requires list names — e.g. expected_output for correctness, context for hallucination.

That's it — restart the dashboard and the new metric is a clickable button. No UI edits needed; app.py renders straight from the registry.
